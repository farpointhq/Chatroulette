    Issue #8: Game lobby & matchmaking UI

    Summary

    Build a polished, dark-themed lobby screen where players can join a matchmaking queue, view online player counts, and transition through states (idle → searching → match-found → error) with smooth animations.

    Root Cause Analysis

    This is a new feature — there is no functional lobby UI today. The project has:

    • Stub components (LandingPage, MatchmakingQueue, OpponentFound, ConnectionError) that all throw Error('Not implemented')
    • A stub Lobby component that only renders a bare heading and status text
    • Type definitions in src/components/lobby/types.ts and src/types/lobby.ts that define the expected props
    • Comprehensive tests in tests/tasks/T8/lobby.test.tsx that describe the full expected behavior
    • A dark-themed base (#1a1a2e background) already set up in index.css

    The current App.tsx passes no-op callbacks to Lobby, so the UI is completely non-functional.

    Proposed Solution

    Implement the Lobby component as a state-driven view switcher that renders the appropriate sub-component based on status. Each sub-component gets its own file with clean, accessible markup:

    1. LandingPage — Title/branding, nickname input, online player count, "Find Match" button (disabled when nickname is empty)
    2. MatchmakingQueue — Animated spinner, "Searching..." text, "Cancel" button, optional estimated wait time
    3. OpponentFound — Opponent info (nickname, rating), assigned color, countdown timer, Accept/Decline buttons
    4. ConnectionError — Error message display with retry button, while still showing the idle UI underneath

    All components use the existing dark theme with polished transitions.

    Test Strategy

    • Unit tests: Run vitest run tests/tasks/T8/lobby.test.tsx
    • All 32 test cases must pass covering: idle state, searching state, match-found state, in-game state, error state, accessibility
    • Edge cases:
    • Empty nickname should disable "Find Match"
    • estimatedWaitTime null should hide wait time display
    • matchFound null during match-found status is an invalid state (type-safe, won't happen)
    • Error state still shows idle UI (nickname input + Find Match button)
    • Visual verification: Open npm run dev and manually cycle through states

    Risks & Mitigations

    Risk Mitigation
    Tests use different prop names than existing types The test file imports from src/components/Lobby (non-existent path). Need to reconcile LobbyProps from types.ts with test expectations. The test expects status, nickname, onlinePlayers, etc. while existing src/types/lobby.ts uses state, playerCount. Use the test's prop names as the source of truth.
    Socket.io integration is out of scope Build with mock setTimeout states in App.tsx. Real socket.io wiring handled in issue #2.
    Accessibility requirements (landmark, label) Tests enforce <main> landmark and labeled input — implement from the start.
    Animation complexity Keep animations pure CSS (no Framer Motion dependency). Use transition and @keyframes for smooth, lightweight effects.
